Datamart: Lakehouse Architecture with Apache Iceberg

A case study on how I design Gold Datamarts by domain/KPI, standardize the data model, and optimize query performance on a Lakehouse + Apache Iceberg architecture (supporting both batch and near‑real-time).

Overview Architecture Dataflow Deliverables View Dashboards →

1) Project overview

This Datamart project focuses on standardizing the data flow from source → landing → bronze/silver → gold to power dashboards/analytics with three goals: correct, fast, and reliable. The gold layer is designed by business domain, with consistent KPI definitions and full traceability.

Problems we solved
  • Fragmented mart logic; KPIs inconsistent across reports
  • Slow queries due to suboptimal partitioning/formats and missing the right aggregations
  • Lack of standardized schema and layered data model, making debugging/lineage difficult
What we built
  • Domain-based gold datamarts (facts/dims) with clearly defined grain
  • Standardized KPI definitions, business rules, mappings, and data contracts
  • Monitoring/validation for data quality with anomaly alerts

2) Architecture overview

The architecture combines lake storage + file formats + a table format (Iceberg) + compute engines (Spark/Trino) + a catalog. This design supports schema evolution, snapshot/version management, and query optimization via partitioning + metadata pruning.

3) Dataflow: source → gold layer

Layering strategy
  • Landing: raw ingestion, preserving the original structure
  • Bronze: minimal schema standardization + naming conventions
  • Silver: clean, dedupe, consolidate (business keys)
  • Gold: datamarts theo use-case/KPI + aggregates

4) Roles & deliverables

Datamart design
  • Translate business needs → logical model (facts/dims), and define grain
  • Define KPIs, business rules, mappings, and documentation
  • Design partitioning/aggregation to optimize query performance & cost
Quality & performance
  • Review/optimize marts: partition, join patterns, compute cost
  • Data checks: completeness, uniqueness, freshness + anomaly signals
  • Set up metric contracts to reduce “definition drift” across teams

5) Impact

Focused on operational stability and improving the quality of analytics delivery.

Business outcomes
  • Reduced refresh latency and improved metric reliability
  • Consistent KPIs across multiple dashboards/reports
  • Enable use-cases: BI, near-realtime, ML pipelines
Operational outcomes
  • Easier debugging thanks to clear layering and lineage
  • Standardized mart templates for faster onboarding
  • Fewer “data drift” issues thanks to checks and alerting

6) Tech stack

Core components
  • Storage: S3 / Azure / GCS
  • Formats: Parquet / Avro
  • Table format: Apache Iceberg
  • Engines: Spark / Trino
Governance & reliability
  • Catalog: Hive / Glue / Nessie
  • Orchestration: Airflow / Dagster (depending on your stack)
  • Quality: expectations + alerts
  • Docs: metric definitions + data contracts
